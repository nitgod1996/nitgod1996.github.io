# 一、 常见的

# 1. typeof、instanceof、constructor、new.target 的区别

这些函数、关键字都可以判断数据类型，总结一下他们的区别

typeof： typeof 主要用于基本数据类型以及函数的判别，对引用数据类型 object 就无法细分其类别了。其以字符串形式返回结果

语法：

  1. typeof+空格+变量/常量
  2. typeof(变量/常量)
  • undefined ：表示值未定义
  • boolean ：布尔值
  • string ：字符串
  • number ：数值
  • object ：对象或 null
  • function ：函数和 class
  • symbol ：符号

instanceof： instanceof 主要是判断后者是否处于前者的原型链上游。

可以用来判断：

  1. 对象实例是否属于该对象（伪类）
  2. 对象（伪类）是否继承于另一个对象（伪类）

<font color=red> instanceof 不适用于原始值（简单类型）</font>

constructor： constructor ：对象的属性，是一个指针，指向创建该对象的构造函数。

不过在 JavaScript 中构造函数也相当于对象（伪类）了，相当于就是返回对象实例属于哪个对象（伪类）

constructor 是属于对象实例的属性

但可以通过原型 prototype.constructor 获取构造函数（因为每个类的原型就是一个原型实例对象）

严格说， constructor 就是属于 prototype 的属性，可以通过 prototype 修改 constructor ，本来是不可枚举，修改后不指定枚举状态的话是默认可枚举

new.target：独属于函数的属性，返回对象实例的构造函数。

  • 如果函数正常调用（没有用 new 实例化）： new.target 返回 undefined
  • 如果 new 实例化了对象： new.target 返回该构造函数

因为 new.target 是函数的装有属性，和 arguments 一样，因此 <font color=red> 只能在函数体内调用 </font>

# 2. 上下文、作用域及垃圾回收

# 2.1 上下文

我的理解是代码执行环境，以函数作用域为单位，包括该执行环境的变量、函数声明、参数等信息

执行上下文主要有全局上下文（window）和函数上下文

  • 函数上下文在所有代码执行完毕会被销毁。
  • 全局上下文在程序关闭时（网页关闭）销毁
  • 全局执行上下文只有一个，函数执行上下文在每次调用函数时候会创建一个新的函数执行上下文。

生命周期

函数调用，当代码执行流进入函数时，该函数调用的上下文（保存该层）压入上下文栈，调用结束弹出上下文（继续该层的执行）。

解释一下：就是在一行行代码中，调用了一个函数，那么把执行到调用函数为止的参数等信息（上下文）先保存到栈里，进入函数进行编译，函数执行完后，再把栈里的这些参数等拿出来继续执行。

# 2.2 垃圾回收

JavaScript 会自动释放内存，只要变量不再被引用，便会被回收

标记清理（最常用）

当变量进入上下文（在该作用域被声明）时，该变量会被加上 “存在于该上下文” 的标记（标记 called 存在于上下文），然后离开时也会被加上 “离开上下文” 的标记。

垃圾回收程序运行时：

  1. 先标记内存中所有变量（认为所有变量都是待删除的）
  2. 遍历所有变量
  3. 找出在上下文中的变量（集合 A）（正在使用的变量）
  4. 找出所有被 A 引用的变量（集合 B）
  5. 去除 A 和 B 的标记
  6. 剩下的带标记的都清理了

几乎所有主流浏览器都使用这种方法清理内存。

标记方法有多种：

  • 当变量进入上下文时，反转某一位
  • 维护 “在上下文中” 和 “不在上下文中” 2 个变量列表
  • 等

引用计数

思路：对每个值记录它被引用的次数

  1. 该值每被一个新的变量引用，引用值 + 1
  2. 引用该值的变量被覆盖 / 删除，引用值 - 1
  3. 内存回收时， if 该变量引用值 ===0，则回收该值

引用计数看上去很简单，但同样问题也很多，比如：

  1. a 和 b 互相引用，则 a 和 b 永远不会被删除，但实际上 a 和 b 可能都要被删除

回收周期

垃圾回收程序周期运行，当垃圾累积到一定数量会影响性能，因此回收周期（时间调度）很重要。

回收频率太低会导致垃圾积压，频率太高会到处小内存片泛滥。

IE7 后，JavaScript 引擎的垃圾回收程序为动态改变分配变量、字面量或数组槽位等阈值。

  • 设一个初始阈值。

  • 如果有一次回收内存不到已分配的 15%，则阈值会翻倍。

  • 如果有一次回收内存达到 85%，则阈值重置为默认值。

（有点像网络拥塞控制的慢开始和快重传）

# 2.3 内存泄露

上面说了，被引用的变量不会被回收内存，而我们有时候可能因为一些失误导致一些不必要的变量一直被引用着。

  • 通过 const 和 let 提升性能： const 和 let 都以块为作用域，比 var 能更早被垃圾回收。

  • 尽可能少地制造隐藏类：即对象实例不要添加新的属性。

    这时候 a1 和 a2 同属于一个类（ Article ）

    这时候 a1 和 a2 还是同属于一个类（ Article ），但他们不属于同一个隐藏类。其中， a2 的隐藏类比 a1 多了一个 author 属性。

    <font color=red> 能够共享隐藏类的对象性能更好，即尽可能少的制造隐藏类 </font>

  • 闭包可能会引起内存泄露：不要在闭包写长期占用的变量引用

# 3. rem 自适应

摘要：rem 是相对于根元素 <html>，这样就意味着，我们只需要在根元素确定一个 px 字号，则可以来算出元素的宽高。

flexible.js 是如何通过 rem 实现自适应的呢？ 通过 rem 与 px 的换算，你可以把设计稿从 px 转到 rem。

简单的理解， rem 就是相对于根元素 <html> 的 font-size 来做计算。而我们的方案中使用 rem 单位，是能轻易的根据 <html> 的 font-size 计算出元素的盒模型大小。而这个特色对我们来说是特别的有益处。

简言之：rem 即是 html 标签的 font-size 的倍数，如下

width 即是 120px。1rem=20px

通过 flexible.js 根据屏幕分辨率自动计算合适的 html 的 font-size ，rem 的大小也就能随着屏幕（窗口）大小随动了。

# 二、 面向对象

# 1. 原型链

# 1.1 原型 prototype

  • prototype 是对象（伪类）的一个属性，继承于 object 对象

  • prototype 是所属对象的一个对象实例，即原型对象

    因此 prototype 都是以 {} 的形式定义的， {} 就是一个对象实例，其默认 constructor 属性指向其对象（伪类），而我们自己创建的 {} 的 constructor 指向 object

  • 构造函数初始化的属性不在 prototype 内

  • prototype 有个 constructor 属性指向构造函数

# 1.2 原型层级（原型链）

可以简单的理解为继承关系，因为 js 的继承是通过原型实现的。关键代码 Child.prototype=new Father(); ，子类的原型是父类的实例

对象实例→对象原型（父类 1 实例 /new 构造函数）→父类 1 原型（父类 2 实例 /new 构造函数）→...→object，这就是一条原型链。

  1. 对象创建了一个实例，当该实例访问一个属性时，

  2. 首先看这个属性是该实例新增的还是在该实力上修改过；

  3. 如果没有，那么看看构造函数有没有初始化这个属性（这层也是相当于实例上的，因为构造函数只有在 new 的时候才会初始化实例和属性）；

  4. 如果还是没有，那么看看对象的原型上有没有这个属性；

  5. 还是没有 ，再看原型对象的原型（父类的原型）有没有初始化这个属性；

  6. 还是没有，再往原型链上游一步步找，直到 object 为止。

# 1.3 原型的 2 大属性：constructor 和_proto_

  • prototype.constructor ：隶属于 prototype 的属性，指向构造函数。
  • prototype.__proto__ ：隶属于 prototype 的属性，指向构造函数的原型对象。即对象的原型

<font color=red> __proto__ 也可以说是指向原型链中的上一级原型。因此__proto__是原型链上至关重要的属性，原型链的往上遍历就是通过 __proto__ 实现的。</font>

# 2. prototype、__proto__和 constructor 的区别

  • prototype ：原型，无需多言，函数做对象类使用时涉及的属性，保存该类的部分（还有部分是保存在构造函数中的）成员函数、成员变量和一些对象类的属性（如 constructor 和 __proto__ 均是 prototype 的属性）
  • prototype.constructor ：隶属于 prototype 的属性，指向构造函数。详见第一章的 1
  • prototype.__proto__ ：隶属于 prototype 的属性，指向构造函数的原型对象。

下面着重讲一下 __proto__ 的性质，及一些相关的对象：

在 JavaScript 原型链中，最底层的是对象实例，其次是对应构造函数的原型对象，再次是父类的原型，一直到 Object，如下所示：

对象实例→对象类。原型→对象类父类。原型→...→Object. 原型→null

因此 __proto__ 也可以说是指向原型链中的上一级原型。

如何分别解释 2 中说法呢：

  1. 指向构造函数的原型对象
  2. 指向原型链中的上一级原型。

JavaScript 中的继承是通过原型链实现的，子类（Child）的原型就是父类（Father）的对象实例，因此以上 2 种说法在逻辑上等价

当访问一个对象的属性时：

  • 如果该对象内部不存在这个属性，那么就会去它的 __proto__ 属性所指向的对象原型里找；

  • 如果还是不存在，则继续往父对象的 __proto__ 属性所指向的那个原型（可以理解为爷爷对象的原型）里找；

  • 直到原型链顶端 null，null 为原型链的终点

所以顺着原型链找就是通过 __proto__ 实现的

# 3. 属性取值优先级

优先级：实例赋值 > 构造函数赋值 > 原型赋值
